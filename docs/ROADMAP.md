# Bootlane Roadmap

Last updated: 2026-06-10

This roadmap is a candidate backlog, not a release promise. It keeps broad post-launch ideas parked with evidence status, review triggers, and next actions before they become implementation work.

Use it with:

## Roadmap Goals

- Protect the 0.1.0 release candidate from broad unproven scope.
- Turn repeated launch feedback into fixture-backed 0.2.0 candidates.
- Keep integrations and write-capable flows outside the default `bootlane check` path until their boundaries are explicit.
- Make out-of-scope ideas easy to close with a consistent local-first rationale.

## Candidate Statuses

| Status | Meaning |
| --- | --- |
| Collecting evidence | The idea is plausible, but maintainers still need reports, fixtures, or workflow examples before planning work. |
| Ready for fixture | The idea is small enough to become a failing fixture or focused test before implementation. |
| Parked | The idea may fit later, but current release quality or evidence is not strong enough to prioritize it. |
| Deferred | The idea needs a larger design boundary and should not enter the current milestone. |
| Closed out of scope | The idea does not fit Bootlane's local-first setup-readiness scope in the default product path. |

## Review Cadence

| Review | When | Required input | Decision |
| --- | --- | --- | --- |
| Post-launch feedback batch review | After the first public feedback batch and then whenever several related reports arrive together. | Grouped issues or discussions mapped to Roadmap Candidate Review categories and evidence status. | Promote repeated fixture-sized work to `Ready for fixture`, keep weak signals in `Collecting evidence`, or close default-path out-of-scope requests. |
| Milestone planning review | Before opening or reshaping a 0.2.0, 0.3.0, or later milestone. | Candidate target, required evidence, owner, user impact, and current release-candidate stability notes. | Only move candidates with evidence and a clear boundary into milestone planning; keep speculative work parked or deferred. |
| Staleness review | When a candidate has no new evidence for a review cycle or its review trigger is no longer current. | Last evidence link, last review date, current target, and whether the request still matches product boundaries. | Refresh the required evidence, downgrade to `Parked`, move to `Deferred`, or close out of scope with rationale. |
| Boundary review | Before planning integrations, new ecosystems, write-capable flows, or anything that may affect the default check path. | Read-only impact, release-safety impact, command boundary, fixture strategy, and docs impact. | Proceed only when default `bootlane check` stays local and read-only; otherwise defer or require an explicit opt-in design. |
| Closeout review | When a candidate is implemented, superseded, duplicated, or no longer fits Bootlane's scope. | Issue or PR link, final status, rationale, and any follow-up candidate if work continues elsewhere. | Close with a documented outcome so the backlog does not accumulate stale release promises. |

## Status Transitions

| From | To | Condition | Required action |
| --- | --- | --- | --- |
| Collecting evidence | Ready for fixture | Repeated reports, a minimal reproduction, or a representative public repository can be turned into a focused fixture. | Record the fixture target, expected finding or report change, and false-positive risk before implementation. |
| Collecting evidence | Parked | The idea remains plausible but lacks repeated evidence or a clear next milestone. | Keep the review trigger explicit and wait for stronger feedback before planning work. |
| Ready for fixture | Collecting evidence | The fixture cannot be minimized, expected behavior is unclear, or false-positive risk is higher than expected. | Capture the uncertainty and ask for narrower evidence before changing checks. |
| Parked | Deferred | The idea needs a larger design boundary, release-safety review, or opt-in command design before it can be planned. | Record the design boundary that must exist before the candidate can be reconsidered. |
| Parked | Closed out of scope | The idea depends on cloud accounts, telemetry, LLMs, vulnerability scanning, network-only behavior, or default-path writes. | Close with local-first setup-readiness rationale and link any optional future discussion if appropriate. |
| Deferred | Parked | A design boundary exists, but the work still lacks evidence or priority for an active milestone. | Update the target and review trigger without starting implementation. |
| Closed out of scope | Collecting evidence | The product scope changes or new evidence reframes the request as local-first setup-readiness work. | Re-run the Roadmap Candidate Review and document why reopening is justified. |

## 0.2.0 Accuracy Work Intake

Use this intake before moving accuracy candidates from roadmap planning into implementation. Accuracy work should stay fixture-first and should improve existing Node and Python setup-readiness behavior before adding new product surfaces.

### Intake Lanes

| Lane | Roadmap candidate | Source | Required fixture | Priority rule | Implementation gate |
| --- | --- | --- | --- | --- | --- |
| False-positive reduction | Real-world accuracy fixture expansion | Confirmed reports where Bootlane reports a finding that should not appear for the project shape. | A minimal fixture or focused unit test that reproduces the noisy finding before the fix. | Prioritize high-confidence checks, common setup patterns, and reports that would make users distrust the first run. | Enter implementation only after the expected absence, severity change, confidence change, or message change is documented. |
| Missing-detection coverage | Real-world accuracy fixture expansion | Reports where Bootlane misses a Node, Python, package-manager, env, runtime, test, README, or CI signal. | A minimal fixture containing the missed signal plus a failing expectation for the intended finding or report field. | Prioritize repeated project shapes, supported Node/Python metadata, and misses that affect run readiness more than score cosmetics. | Enter implementation only after the signal source, expected check ID or report field, and existing happy-path impact are known. |
| Env var precision | Env var scanning precision pass | Reports about env var false positives, missed env access, placeholder handling, comments, strings, or confusing env guidance. | Focused env fixtures that cover source access, comments, string examples, placeholders, and .env.example expectations. | Prioritize cases that affect supported access patterns before adding new syntax or broader scanning rules. | Enter implementation only after redaction behavior, expected env names, and non-source ignore behavior are covered. |
| Package-manager command recognition | Real-world accuracy fixture expansion | Reports where README or CI command recognition misses npm, pnpm, Yarn, Bun, pip, uv, Poetry, or Pipenv setup patterns. | A minimal README or workflow fixture with the command form plus expected install, run, or test signal behavior. | Prioritize package managers already supported in 0.1.0 and command forms repeated across more than one report. | Enter implementation only after README and CI behavior can share or intentionally separate the command recognition rule. |
| Report clarity and scoring adjustment | Real-world accuracy fixture expansion | Reports where the detection is technically correct but severity, confidence, score impact, or wording misleads users. | A snapshot or focused reporter/scoring test that shows the confusing output before the change. | Prioritize wording and severity fixes over detector rewrites when the underlying detection is correct. | Enter implementation only after JSON schema stability, check ID stability, and Markdown example impact are reviewed. |

### Prioritization Rules

| Priority | Applies to | Required evidence | Action |
| --- | --- | --- | --- |
| P0 trust regression | False positives that make a supported project look unsafe, broken, or impossible to run when the evidence says otherwise. | Reproduction command, finding ID, minimal fixture, expected output, and why the current result damages first-run trust. | Fix before adding broader detection or new roadmap work. |
| P1 repeated supported miss | Missing detections repeated across supported Node or Python project shapes. | Two related reports or one representative public repo plus a minimized fixture and expected signal behavior. | Schedule for 0.2.0 once existing happy-path fixtures remain stable. |
| P2 precision polish | Env, README, CI, package-manager, scoring, or wording issues that are correctable with focused fixtures. | Focused fixture or snapshot plus expected severity, confidence, score, or message change. | Batch with nearby accuracy work to avoid churn across check IDs and report examples. |
| P3 speculative expansion | Requests for new syntax, ecosystem expansion, integrations, or broad heuristics without repeated evidence. | Roadmap candidate category, review trigger, and reason the work is not yet fixture-ready. | Keep in the Roadmap backlog instead of entering implementation. |

### Fixture Case Matrix

Use this matrix after the intake fields are known and before implementation starts. It maps each accuracy lane to the minimum fixture shape, expected report diff, and happy-path fixture protection that should be explicit in the issue or PR.

| Lane | Case type | Minimal files | Expected report diff | Protects happy path | Ready when |
| --- | --- | --- | --- | --- | --- |

| Missing-detection coverage | Missed supported signal fixture | Smallest fixture containing the missed Node, Python, package-manager, env, runtime, test, README, or CI signal. | A new finding, check ID, report field, or command signal appears with the documented severity, confidence, and score impact. | `examples/fixtures/node-good` and `examples/fixtures/python-good` stay false-positive-free for the touched detector or shared signal helper. | The expected report diff is asserted in a focused test and no supported happy-path fixture gains a new finding. |
| Env var precision | Env access and ignore fixture | Focused source, comment, string-example, and `.env.example` files that isolate the reported env access or placeholder behavior. | Env names are detected, ignored, or redacted exactly as expected without exposing full secret values. | Node and Python happy-path fixtures keep their env example status and secret redaction behavior stable. | Source access, non-source ignores, placeholder handling, and redaction behavior are covered by fixtures or snapshots. |

### Accuracy Work Intake Closure Criteria


| Outcome | Applies to | Required evidence | Close when |
| --- | --- | --- | --- |
| Fixture-backed fix merged | Confirmed false positives, missing detections, env precision issues, command recognition misses, and report clarity fixes. | Accuracy lane, Accuracy priority, Fixture target, Expected output, Happy-path impact, and a failing fixture or focused test that passes after the fix. | The fix merges, relevant docs or examples are updated, and existing Node/Python happy-path fixtures remain stable. |
| Needs more fixture evidence | Reports with plausible accuracy impact but missing reproduction details, fixture target, expected output, or happy-path impact. | A maintainer comment naming the missing fixture intake fields and the narrow evidence needed next. | The reporter provides the missing evidence, or the issue is marked `needs-more-info` and closed after the follow-up window. |
| Moved to roadmap backlog | Accuracy-adjacent requests that are too broad, speculative, repeated but not fixture-ready, or better handled as 0.3.0+ planning. | Roadmap candidate, evidence status, review trigger, and rationale for not entering 0.2.0 implementation. | The Roadmap entry is updated and the issue links the candidate, review cadence, or status transition that will revisit it. |
| Closed out of scope | Requests that depend on cloud accounts, telemetry, LLMs, vulnerability scanning, network-only checks, or default-path writes. | Local-first setup-readiness rationale plus any optional future discussion link if appropriate. | The issue explains why the request is outside the default `bootlane check` path and no fixture-backed action remains. |
| Accuracy PR ready for review | Pull requests that implement 0.2.0 accuracy work. | PR template confirms Accuracy lane, Accuracy priority, Fixture target, Expected output, Happy-path impact, read-only behavior, and relevant docs or report updates. | Review can start because fixture evidence, schema/check ID impact, and happy-path impact are explicit. |

## Backlog Maintenance Rules

- Every roadmap candidate must use a category from the [Roadmap Candidate Review](TRIAGE.md#roadmap-candidate-review).
- Every candidate must have a target, evidence status, required evidence, review trigger, and next action.
- Every review cycle should use the Review Cadence table before changing targets or evidence status.
- Every evidence status change should match a Status Transitions row and leave a short rationale in the issue, PR, or roadmap update.
- `0.2.0` candidates should be fixture-first accuracy work from real feedback.
